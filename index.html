Introduction

Adulting is real. The more you get older, the more payments you have. This is true to my case where I and my husband have a lot of bills payments such as health and life insurance, condominium investment, house rent, telco bills, remittance to the family in the Philippines,among others. As a result, due date of bills payments is often forgotten.

Planning
  1. Statement of the problem

    We need to check our bank app again and scrolling through our emails for history to check whether we already paid or not, and this is very time consuming. This caused us a lot of headache. As a matter of fact, we could have used the Note app but it cannot trace any relationsship of the bills such as the history on monthly basis.

  2. Solution Strategy

    As a solution, I developed this app called Due List. This is somewhat similar to bank application but fewer features. Once the payment or remittance is done, we should input it in the Due List app right away.

  3. Project Schedule

    Since I have a full-time job, I can allot only at least 2 hours per day on developing this project. It took me almost two months to complete it.

Requirement Analysis
  1. Functional Requirement
    1. The system should be able to store data of the user, due list and transactions.
    2. The user should be able to register and log-in.
    3. The user can update the password in case he or she forgets it.
    4. The user should be able to add, update and delete the due list.
    5. The user should be able to add transaction under the bills name.
    6. The user should be able to view past transactions and select the range of months.
    7. The user should be able to add to the Google Calendar of each bills name registered.
  2. Non-Functional Requirement
    1. Platform: The application should work in both desktop and mobile devices.
    2. Database: The data should be stored in the cloud so it would be real-time update.
    3. Performance: The system should be able to handle multiple concurrent users to use the system without breakdown.

Software Design
  1. Data Design

    The Relational Model is created using Draw.io a open source software.


      Description of the Model
    • A user can have one or more due lists.
    • A due list can have one or more transactions.
    • A transaction can only be owned by one due list.

      Description of the Tables
    • User
      • _id : unique number, objectid which is auto-generated
      • username : string
      • password : string
      • name : string
      • email : string
    • DueList
      • _id : unique number, objectid which is auto-generated
      • user_id : string (Foreign key from user _id attribute)
      • bills_name : string
      • frequency: string ({0:Monthly,1:Quarterly,2:Sem-annually,3: Annually})
      • benefeciary_name : string
      • scheduled_day : Number (1-31)
      • amount : Number
      • Currency : String (SGD or PHP)
      • txn : Array (child of due list table)
    • txn
      • _id : unique number, objectid which is auto-generated
      • date_paid : Date (Default: Date.now)
      • amount:Number
      • Currency : String (SGD or PHP)
      • paid_by : String
      • mode_payment : String

  3. Procedural Design
      Registration
    1. User will click the register button.
    2. System will display the registration interface.
    3. User will input the data.
    4. System will pre-evaluate the data provided.
    5. User will click the register now.
    6. System will prompt the user if want to proceed.
    7. The password will be hashed and the details will be registered to the user table.
    8. The system will automatically redirect to login page.

      Reset Password
    1. User will click the forgot button.
    2. Sytem will display the reset password interface.
    3. User will input the registered email address.
    4. System will send token (4-character) to the email address provided.
    5. User will type the token received.
    6. System will verify if the token inputted is the same as the one sent to the email.
      1. If the token is the same, it will ask the user to type the new password.
      2. The system will hashed the password and will update the new password associated to the email address provided.
      3. Once successful, the system will automatically redirect to login page.

      Log-in
    1. User will type the username and password.
    2. The system will check if the username exists in the database.
      1. If the username exists, it will compare the password typed and the password in the database using bycrypt API.
      2. If it matches, it will redirect to the homepage.
      3. Else, system will automatically redirect to login page.

      Create or Add Bill's Name
    1. User will click the add button.
    2. System will display the Add Bill Interface.
    3. User will input the details: Bill's name, Benefeciary name, Frequency, scheduled day, amount to pay, and currency.
    4. System will prompt if the bill will be added.
    5. If the user will click yes, the bill will be added to the duelist table.
    6. The system will display alert of successful creation of bill.
    7. Else, it will stay in the same page.

      Delete Bill's Name
    1. In the homepage, user will click the X (delete) button.
    2. System will ask the user if he/she will delete the bill.
    3. If the user will click yes, the bill and all its transactions will be deleted as well.

      Search Bill's Name
    1. In the homepage, user will type the name of bill on the search textbox.
    2. When the user will unfocus the textbox, the system will automatically search the bills that matches the keyword in the textbox.

      Update Bill's details
    1. In the homepage, user will click the See Detail button.
    2. The system will display the Edit Bill Details interface.
    3. User will click the Edit button.
    4. System will show the Clear and Update buttons.
    5. User will update the details.
    6. If the update button is clicked, the system will update the details to the duelist table.
    7. If the clear button is clicked, the details in the textboxes wil be cleared.

      Add transaction
    1. In the homepage, user will click the See Detail button.
    2. The system will display the Edit Bill Details interface.
    3. User will click the Add Transaction button (card icon).
    4. System will display the Add Transaction interface.
    5. User will input the details such as as data, amount paid, currency, paid by and mode of payment.
    6. User will click the Add button.
    7. System will ask the user if he/she will add the transaction since it cannot be deleted or updated any more.
    8. When the user will click yes, the transaction will be inserted to the duelist details.
    9. Else, it will stay in the same page.

      Add to Google Calendar
    1. In the homepage, user will click the Add to Calendar button.
    2. Sytem will display the Add to Google Calendar interface.
    3. User has the options to select the number of years (1, 2 or 3 years) where the event is to be added in the calendar.
    4. User will click the Add to Google Calendar button.
    5. System will ask the user if he/she wanted to continue.
    6. If the user will click yes, the system will display the google authentication.
    7. User will follow the instruction.
    8. If he/she allowed to access the google calendar, the event will be added to the calendar associated to the email he/she is logged in.

      View Transaction History
    1. In the History page, the transactions for the past three months will be displayed in the screen.
    2. When the user click the month name, the system will display the transaction related to that month.
    3. When the user select the months to display the transaction, the system will automatically filter the dates associated in the month range.

      Log-out
    1. User will click the Settings tab.
    2. System will display the Settings interface.
    3. User will click the log-out button.
    4. System will redirect to the log-in page.

So I googled which one best for the beginner programmer. I read some topics regarding the difference between VueJS, ReactJs and AngularJS. Some says that for beginner like me, VueJS is a better start to learn. Without hesitation, I tried it by getting my hands dirty and true enough I was in love with Vue. After that, I was thinking to develop my own app which could be beneficial to me (first).

Software Deployment
  1. Mobile and Desktop Platform

    The server-side and client-side is deployed in the Heroku. The following code shows how to deploy to Heroku.

    heroku create bills-payment-lists
    heroku git:remote --app bills-payment-lists
    git commit -am "remove dist fromgit ignore"
    git push heroku master

  2. Android Platform

    In order for the ionic/vue to be able to deploy to the android platform, we should use Capacitor. It allows to run the application across desktop, mobile and android platforms.

    The following code is done in order to add android platform.

    npm run build
    npx cap copy
    npx cap add android
    npx cap open android
